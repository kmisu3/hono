`docs/design-documents.md`を基準に、必要な設計資料だけを管理する。

1. Issue、依頼内容、既存実装、`docs/architecture.md`、関連するAPI・DB契約を確認する。
2. `docs/design-documents.md`の基準で、機能設計書とADRの要否をそれぞれ判断する。
3. 作成・更新する場合:
   - 機能設計書は`docs/designs/_template.md`を使用する。
   - ADRは`docs/decisions/_template.md`を使用し、1件の重要な判断だけを記録する。
   - コードやOpenAPI、テーブル定義書で管理する値を重複して記載せず、正となる資料へリンクする。
   - 根拠のない要件や将来案を追加しない。
4. レビューだけを依頼された場合はファイルを変更せず、次を確認する。
   - 必要な機能設計書・ADRが存在するか。
   - Issue、設計資料、OpenAPI、DB定義、実装、テストが矛盾していないか。
   - ADRが機能固有の小さな判断や作業ログで増えていないか。
   - 設計書がコードの説明を重複していないか。
5. 設計資料の変更は関連する実装と同じコミットに含める。

設計資料が不要と判断した場合は、不要な理由を簡潔に報告し、形式的な文書は作成しない。
