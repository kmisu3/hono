# Honoを最小限の責務分離で利用する

- 状態: 採用
- 決定日: 2026-06-12

## 背景

小〜中規模のREST APIを、Honoの軽量さと型推論を活かしながら変更しやすく保つ必要がある。
初期段階から多層構造やDIコンテナを導入すると、単純な変更にも不要な実装が増える。

## 決定

HonoのルートハンドラーをHTTP境界として使用し、複雑になった業務処理だけをServiceへ
切り出す。DBアクセスはRepositoryへ閉じ込め、依存は関数引数で明示的に注入する。

## 選択肢

- 最小限の責務分離: 現在の規模に合い、必要になった時点で段階的に分割できるため採用。
- Controller・Service・Repositoryを常に作成: 単純なCRUDでも構造が増えるため不採用。
- DIコンテナ: 現在の依存関係では導入効果より複雑さが上回るため不採用。

## 影響

- 単純な処理はルートからRepositoryを直接利用できる。
- 業務ルールや複数処理の調整が増えた場合はServiceを追加する。
- Hono固有の型や処理をService・Repositoryへ持ち込まない。
